# 2026-04-04 动态站点与 Provider 相关问题修复总结

## 背景
本轮修复主要围绕两类问题展开：

1. **Provider / 自定义 OpenAI-compatible 配置诊断能力不足**
2. **动态站点（尤其是 X / Twitter）上的翻译、滚动、恢复、增量更新体验不稳定**

目标不是只修单点 bug，而是把系统从“更适合静态页面的一次性批量翻译”逐步调整为“更适合动态站点的渐进式翻译”。

---

## 一、custom OpenAI-compatible 测试连接只显示 400

### 现象
选择自定义 OpenAI-compatible Provider 后，测试连接失败时界面只显示：

- `服务器返回错误: 400`

无法判断到底是：

- endpoint 不对
- model 不对
- API key 不对
- 请求体参数不兼容

### 分析
对比多供应商改造前后的实现后确认：

- **Background 中真正发请求的逻辑基本没变**
- 多供应商改动主要集中在 **Popup 表单层**
- 真实问题在于：**测试连接时没有把服务端响应体展示出来**

### 方案
在 `src/background/index.ts` 中增强 `handleTestConnection()`：

- 失败时读取响应体
- 优先提取这些字段：
  - `error.message`
  - `error.type`
  - `error.code`
  - `message`
  - `detail`
- 在 popup 中直接显示详细错误信息

同时在 `src/popup/style.css` 中让错误结果支持多行换行显示。

### 效果
测试连接失败时，不再只看到 `400`，而是能看到服务端返回的具体错误信息，便于快速定位配置问题。

## 二、悬浮进度数字不可信，1/2 后突然完成

### 现象
页面右下角悬浮进度经常表现为：

- 先显示 `1`
- 再显示 `2`
- 然后突然变成成功

用户感知上不稳定，也无法理解数字含义。

### 根因
此前系统把两种不同语义混在了一起：

- **显示进度**：按 API 批次返回数推进
- **完成判定**：按真正渲染到页面的批次数完成

在并发请求乱序返回时：

- 数字会提前变化
- 页面实际内容还没完全显示
- 最后又会突然完成

### 设计决策
最终采用的原则是：

> **用户可见进度应当表示“已经真正显示在页面上的内容块 / 总内容块”**

而不是内部实现的 batch 数。

### 方案
重构 `src/content/translator.ts`、`src/content/index.ts`、`src/background/index.ts`：

- 进度单位改为 **内容块（block）**
- 长文本即使拆成多个 chunk 请求，也只算 **1 个 block**
- 只有译文真正插入页面后，才推进 completed
- popup 顶部状态和页面悬浮进度统一语义
- Background 不再自己根据请求完成数推断进度，而是由 content script 在实际渲染后上报

### 效果
进度变成：

- `已渲染块数 / 总块数`

不再暴露内部 batch 细节，用户感知更可信。

## 三、翻译完成后滚动导致译文被清空

### 现象
翻译完成后页面滚动，原有译文会被清空。

### 根因
这不是翻译结果丢失，而是 SPA 导航误判：

- 某些站点在滚动时会更新 URL / hash / replaceState
- 系统把这种变化误判成“页面导航”
- 然后执行：
  - `injector.removeAll()`
  - `translator.resetState()`

从而主动清空了译文。

### 方案
在 `src/content/index.ts` 中调整导航判断：

1. **删除翻译完成后自动滚动到第一个译文块的逻辑**
2. **删除显示译文时自动滚动的逻辑**
3. 导航判断只看：
   - `origin`
   - `pathname`
   - `search`
4. 忽略：
   - `hash`
   - scroll 引起的 URL 微变化

### 效果
滚动或 hash 变化不再触发误清空译文。

## 四、滚回已翻译区域时页面会“跳一下”

### 现象
用户先滚到下方，再滚回上方已翻译内容区域时，页面会轻微跳动。

### 根因
原实现中：

- `MutationObserver` 一旦发现译文节点被删除
- 就立即调用 `restoreRemovedTranslations()`
- 重新把译文插入文档流
- 导致布局变化和滚动锚点重新计算

在 React / 虚拟列表 / 动态流站点中，这种“即时补回”非常容易引发跳动。

### 调研参考：FluentRead
调研 `../FluentRead` 后发现其关键策略为：

- 双语译文通常**追加在原节点内部**
- 新内容通过 `IntersectionObserver` 懒翻译
- **不采用我们的“译文被删就立刻 restore”策略**

这个结论对我们后续修复方向影响很大。

### 方案
借鉴 FluentRead，但保留当前项目的安全约束（继续使用 `textContent`，避免 `innerHTML` 风险）：

#### 1. 插入策略调整
在 `src/content/injector.ts` 中：

- 对大多数块级内容，改为**把译文追加到原节点内部**
- 不再优先插成原节点后的兄弟节点

这样可以减少 sibling insertion 带来的结构性重排。

#### 2. 恢复策略调整
在 `src/content/index.ts` 中：

- 不再一发现译文被删就立即恢复
- 先记录被删的译文 ID
- 等用户**滚动停止**后，再只恢复**视口附近**的译文
- 一次恢复少量内容，避免大面积布局变化

#### 3. 滚动补偿
恢复前记录一个视口锚点位置，恢复后按偏移量进行 `scrollBy` 补偿，尽量减小体感跳动。

### 效果
回滚查看已翻译区域时，页面抖动明显降低。

## 五、X 上点击 Show more 后内容没有重新翻译

### 现象
在 X（Twitter）页面，点击 `Show more` 后：

- 同一个 tweet block 内容变长了
- 但扩展没有重新翻译该 block

### 根因
此前的跳过条件过于粗糙：

- 只要某个 Element 已经被翻译过
- 就直接跳过

但在 X 上，点击 `Show more` 后通常还是同一个元素，只是其文本内容变了。

于是系统错误地把它当成“已翻译，无需处理”。

### 方案
在 `src/content/translator.ts` 中引入更精确的判断：

- 为每个已翻译元素记录**上次翻译时的源文本**
- 下一次扫描时：
  - 如果元素相同且文本相同 → 跳过
  - 如果元素相同但文本不同 → 重新翻译

同时在 `src/content/extractor.ts` 中：

- 提取原文时先移除 `.nt-translation / [data-nt]`
- 避免已注入译文污染原文提取结果

并在 `src/content/index.ts` 中让 `MutationObserver` 监听：

- `childList: true`
- `characterData: true`

从而覆盖“节点新增”和“文本直接变更”两种情况。

### 效果
X 上点击 `Show more` 后，变长的 tweet 内容会被重新翻译并覆盖更新。

## 六、X 首次翻译时页面会跳到后面/最后面

### 现象
在 X 页面首次点击“翻译全文”时，页面可能会明显跳到后方，甚至像被推到最后面。

### 根因
这不是自动滚动残留，而是：

- X 是动态流 / 虚拟列表页面
- 首次把整列内容都翻译，会让大量 tweet 高度同时变化
- 触发 X 自己的滚动锚定 / 列表位置重算
- 导致页面被“推走”

### 设计决策
对 X 这种站点，继续做“首次全量翻译”再去补 scroll，并不是最佳方案。

最佳方案是：

> **首次只翻译视口附近内容，其他内容滚到附近后再懒翻译**

### 方案
在 `src/content/index.ts` 和 `src/content/translator.ts` 中实现：

- 对 `x.com` 启用 **viewport lazy translation**
- 首次翻译只处理：
  - 当前视口附近的内容块
- 用户滚动后，滚动停下来时再翻译新进入视口附近的块
- `Show more` 触发的同块内容变化仍支持局部重翻
- 通过 `includeElement` 过滤器限制某次翻译只处理视口附近元素

### 效果
X 上首次翻译不再试图瞬间扩张整列内容，大幅降低页面直接跳走的概率。

## 总体设计思路
本轮修复没有停留在“看到哪里坏了就补哪里”，而是逐步形成了以下设计原则：

### 1. 用户可见状态必须是真实语义
- 进度条显示“已渲染内容块 / 总内容块”
- 不直接暴露内部 batch 实现

### 2. 动态站点必须采用更保守的 DOM 策略
- 不因 hash 或滚动导致的 URL 更新就清空译文
- 不在滚动过程中即时恢复/重翻内容
- 减少大面积同步 DOM 改写

### 3. 借鉴 FluentRead 的动态站点策略，但保留当前项目的安全约束
借鉴的点：
- 视口懒翻译
- 原节点内部追加译文
- 更保守的恢复策略

保留的点：
- 继续用 `textContent` 注入译文
- 不依赖 `innerHTML` 生成译文内容
- 继续保证更清晰的状态同步机制

---

## 最终结果
本轮修复后，系统从更偏向“静态页面一次性翻译”的模型，演进为更适合动态站点的模型：

- Provider 错误更可诊断
- 进度语义更可信
- 滚动不再误清空译文
- 回滚查看内容更稳定
- X 的 `Show more` 支持局部重翻
- X 首次翻译改为视口懒翻译，减少页面跳转与重排问题

这一系列修改构成了后续继续优化动态站点兼容性的基础。
